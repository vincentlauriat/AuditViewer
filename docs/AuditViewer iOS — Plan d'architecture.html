§1

Contexte et périmètre

AuditViewer Mac est une app SwiftUI (Swift 6, macOS 15, SwiftPM) qui affiche et pilote les dossiers d'audit produits par le skill audit-report de Claude Code. L'app iOS/iPadOS cible la lecture et la navigation dans ces mêmes dossiers — sans capacité à lancer un audit (pas de Process ni de NSOpenPanel sur iOS).

Le contrat de données est stable (contrat machine v1) : le skill écrit des fichiers .md numérotés, des fichiers JSON préfixés _, et un flux d'événements _events.jsonl. L'app iOS lit exactement ces mêmes artefacts, qu'ils viennent d'iCloud Drive, d'un partage AirDrop, ou d'un Files provider tiers.

Hypothèse d'accès aux fichiers L'app iOS accède aux dossiers d'audit via le File System API standard (UIDocumentPickerViewController ou URL bookmark de sécurité). La racine Research est stockée dans le Keychain, comme sur Mac. Aucun serveur requis.
Hors périmètre v1 iOS Lancer un audit (claude -p …), surveiller _events.jsonl en temps réel (kqueue indisponible sur iOS — polling uniquement), exporter en DOCX (pandoc absent). Ces features peuvent arriver en v2.
§2

Format de données — contrat v1

Chaque dossier d'audit est nommé audit-<slug> (slug déterministe : NFKD → ASCII → non-alphanum remplacé par -, trim, minuscules). Les fichiers internes se répartissent en deux familles :

Répertoire racine (Research Root — Keychain)
~/Documents/Research/ ou chemin personnalisé
Dossiers d'audit (nommage déterministe)
audit-apple-inc/ audit-openai/ audit-notion/ …
Contenu d'un dossier (skill audit-report v1)
00_RESUME_EXECUTIF.md 01_HISTORIQUE.md → 11_RAPPORT_COMPLET.md
_manifest.json _recon.json _sources.json _data.json _events.jsonl

§4

Événements live — _events.jsonl

Le skill écrit une ligne JSON par événement dans _events.jsonl. Sur Mac, ce fichier est surveillé via kqueue (DispatchSourceFileSystemObject). Sur iOS, il faudra utiliser un timer de polling (suggéré : 1 s pendant un audit actif, 0 autrement) via FileHandle ou NSFileCoordinator.
